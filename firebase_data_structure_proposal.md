# بنية بيانات Firebase المقترحة لميزة معرض الوسائط

لتحقيق ميزة معرض الوسائط الديناميكية بناءً على حالة توثيق المستخدم، سنحتاج إلى تعديل بنية بيانات Firebase الحالية. الهدف هو تخزين معلومات الوسائط، حالة التوثيق، والإشعارات بطريقة منظمة وفعالة.

## 1. مسار `users/{uid}/gallery`

سيحتوي هذا المسار على قائمة بالوسائط (صور وفيديوهات) التي رفعها المستخدم. كل وسيط سيكون له مفتاح فريد (Push ID) وسيحتوي على البيانات التالية:

```json
{
  "users": {
    "[user_uid]": {
      "gallery": {
        "[media_id_1]": {
          "url": "https://res.cloudinary.com/your_cloud_name/image/upload/v12345/public_id_1.jpg",
          "type": "image", // أو "video"
          "timestamp": 1678886400000, // وقت الرفع (ميلي ثانية)
          "cloudinary_public_id": "public_id_1", // معرف الوسيط في Cloudinary للحذف السهل
          "uploaded_at": "2026-07-12T10:00:00Z" // تاريخ الرفع بتنسيق ISO 8601
        },
        "[media_id_2]": {
          "url": "https://res.cloudinary.com/your_cloud_name/video/upload/v12345/public_id_2.mp4",
          "type": "video",
          "timestamp": 1678887000000,
          "cloudinary_public_id": "public_id_2",
          "uploaded_at": "2026-07-12T10:10:00Z"
        }
      },
      // ... بيانات المستخدم الأخرى
    }
  }
}
```

**ملاحظات:**
*   `cloudinary_public_id`: ضروري لحذف الوسائط من Cloudinary عند انتهاء صلاحيتها أو حذفها يدويًا.
*   `uploaded_at`: يمكن استخدامه لتتبع عمر الوسيط لتطبيق قواعد الحذف التلقائي.

## 2. مسار `users/{uid}/verification`

سيحتوي هذا المسار على معلومات توثيق المستخدم، والتي ستحدد عدد الوسائط المسموح بها وفترة الصلاحية.

```json
{
  "users": {
    "[user_uid]": {
      "verification": {
        "status": "monthly", // "none", "weekly", "monthly", "yearly"
        "startDate": 1678886400000, // تاريخ بدء التوثيق (ميلي ثانية)
        "endDate": 1681478400000,   // تاريخ انتهاء التوثيق (ميلي ثانية)
        "lastChecked": 1678972800000 // آخر مرة تم فيها التحقق من التوثيق (لتجنب الإشعارات المتكررة)
      },
      // ... بيانات المستخدم الأخرى
    }
  }
}
```

**ملاحظات:**
*   `status`: يحدد نوع التوثيق وبالتالي الحد الأقصى للوسائط.
*   `startDate` و `endDate`: لتحديد فترة صلاحية التوثيق.
*   `lastChecked`: يمكن استخدامه لتتبع متى تم إرسال آخر إشعار للمستخدم بخصوص توثيقه.

## 3. مسار `users/{uid}/notifications`

سيحتوي هذا المسار على قائمة بالإشعارات الخاصة بالمستخدم، والتي يمكن عرضها في واجهة المستخدم.

```json
{
  "users": {
    "[user_uid]": {
      "notifications": {
        "[notification_id_1]": {
          "message": "تبقى يومان على انتهاء توثيقك الأسبوعي.",
          "timestamp": 1678972800000,
          "read": false,
          "type": "verification_reminder" // لتصنيف الإشعارات
        },
        "[notification_id_2]": {
          "message": "تم حذف صورة واحدة من معرضك بسبب انتهاء توثيقك.",
          "timestamp": 1679059200000,
          "read": false,
          "type": "media_deletion"
        }
      },
      // ... بيانات المستخدم الأخرى
    }
  }
}
```

**ملاحظات:**
*   `read`: لتتبع ما إذا كان المستخدم قد قرأ الإشعار أم لا.
*   `type`: لتصنيف الإشعارات وعرضها بشكل مناسب.

## 4. قواعد أمان Firebase (Firebase Security Rules)

يجب تحديث قواعد أمان Firebase لضمان أن المستخدمين يمكنهم فقط قراءة وكتابة بياناتهم الخاصة في هذه المسارات.

```json
{
  "rules": {
    "users": {
      "$uid": {
        // يمكن للمستخدم قراءة وكتابة بياناته الخاصة
        ".read": "auth.uid === $uid",
        ".write": "auth.uid === $uid",
        "gallery": {
          // يمكن للمستخدم قراءة وكتابة وسائطه الخاصة
          ".read": "auth.uid === $uid",
          ".write": "auth.uid === $uid"
        },
        "verification": {
          // يمكن للمستخدم قراءة حالة توثيقه
          ".read": "auth.uid === $uid",
          // الكتابة قد تكون مقيدة أكثر، مثلاً من خلال وظيفة سحابية (Cloud Function)
          // عند شراء توثيق جديد، أو يمكن للمستخدم تحديث بعض الحقول بنفسه
          ".write": "auth.uid === $uid"
        },
        "notifications": {
          // يمكن للمستخدم قراءة إشعاراته الخاصة
          ".read": "auth.uid === $uid",
          // الكتابة ستكون عادة من جانب الخادم (Cloud Function) أو من خلال التطبيق نفسه
          ".write": "auth.uid === $uid"
        }
      }
    }
  }
}
```

**ملاحظة هامة:** قواعد الأمان أعلاه هي مثال مبسط. في بيئة إنتاج حقيقية، قد تحتاج إلى قواعد أكثر تعقيدًا، خاصة للكتابة في مسارات `verification` و `notifications`، حيث قد ترغب في أن يتم ذلك فقط من خلال وظائف سحابية (Cloud Functions) لضمان سلامة البيانات ومنع التلاعب. على سبيل المثال، عند شراء توثيق، يجب أن تقوم وظيفة سحابية بتحديث `verification` وليس العميل مباشرة.

## 5. تكامل Cloudinary

عند رفع الوسائط، يجب استخدام Cloudinary API لرفع الملفات. بعد الرفع الناجح، سيتم تخزين `url` و `cloudinary_public_id` في Firebase. عند حذف وسيط، يجب استدعاء Cloudinary API لحذف الملف باستخدام `cloudinary_public_id`.
